## 1. 本轮改动点总览

本轮主题为「**做减法**」：页面从 v0.4 的多区块堆叠收敛为「左侧目录导航 + 右侧 9 个模块」的两栏指挥台结构，只保留与 PIP 绩效目标直接相关的内容。

- **两栏布局**：左侧 240px 固定目录（玻璃面板、scroll-spy 滚动高亮、底部周更时间），右侧主区依次排布 9 个模块；≤900px 时侧栏折叠为汉堡抽屉，点导航后自动收起。
- **暗色单主题**：删除日间（light）主题与 Day/Night 切换，固定暗色「业务战情室」（`<html data-theme="dark">`），样式收敛为单套 CSS 变量，无主题闪烁与双主题维护成本。
- **9 模块收敛**：01 执行摘要 / 02 KPI 概览 / 03 时间推进图 / 04 依赖关系图 / 05 主线任务进度 / 06 工作 Pipeline / 07 本周待办 / 08 阻塞事项 / 09 周更记录。目录 9 项与模块锚点一一对应。
- **数据源 9→7**：加载的 JSON 收敛为 `kpi` / `gantt` / `roadmap` / `pipeline` / `todo` / `milestones` / `weekly-log` 共 7 个；4 个弃用文件保留磁盘但不加载（见第 7 节）。

## 2. 删减模块清单与删减逻辑

删减原则一句话：**只保留与绩效目标直接相关的内容**；同一事实不在两处重复展示；执行工具（Kimi）不作为汇报内容。

| 删减项 | 删减逻辑 |
| --- | --- |
| 日间主题与切换 | 双主题意味着每个组件两套配色回归成本；看板实际只在工作汇报场景使用，固定暗色战情室风格后样式与 QA 成本减半，也消除了主题闪烁与 localStorage 记忆逻辑。 |
| CRM 漏斗独立区块 | 与「绩效目标」无直接增量：漏斗表达的客户分层/转化已由 KPI「客户Pipeline」迷你漏斗卡与 pipeline.json 状态字段承载；真实 CRM 明细按脱敏规范本就不入库，独立区块只剩占位示例数据，容易误导读者。 |
| Live Task Progress 独立区块 | 任务级实时进度与甘特子任务、Pipeline 卡片完全同源（都来自同一批任务的状态），属同一事实第三处展示。 |
| 任务树独立区块 | 树形明细与「6 主线可折叠甘特」信息重叠：甘特主线 children 已能表达「主线 → 子任务」的层级与进度，且多了时间维度；独立任务树被甘特展开态取代。 |
| 里程碑独立区块 | 里程碑不是消失而是降维：执行摘要「下一里程碑」卡已自动取 milestones.json 最近未完成节点，独立区块只剩列表式重复。 |
| Hero 拓扑区 | 页首拓扑与 04 依赖关系图功能重叠（都是节点 + 链路表达），且 Hero 占首屏却不含可操作信息；保留信息更完整的依赖图，首屏让给执行摘要。 |
| 私密部署独立内容 | 部署进度属于项目后勤而非绩效内容；访问控制收敛为依赖图支线 B 上的一个节点（「访问权限控制」），正文不再展开部署方案与宣传性描述，操作细节留在 `docs/04`、`docs/05`。 |
| Kimi 任务内容 | Kimi 只是 AI 辅助执行方式，不是工作成果本身；绩效看板回答「事进展如何」，不回答「用什么工具做的」。页面、数据文件与代码中的 owner/责任人字段一律不含 Kimi。 |

## 3. 甘特图重构逻辑

- **仅 6 条主线**：甘特从 v0.4 的 9 根扁平任务条改为 `gantt.json` 的 L1–L6 六条业务主线（业务手册/FAQ、BD 计划文档、设计交付包、客户 Pipeline 建档、注册/KYC/首单推进、周报/CRIB 复盘），主线即绩效口径，不再平铺零散任务条。
- **可展开子任务**：每条主线携带 `children[]` 子任务；点击左列主线名（`aria-expanded` 联动）展开 / 折叠，子任务条更细、同色浅色（opacity .75），悬浮详情含状态·进度 / 负责人 / 起止 / 所属主线。
- **默认折叠**：六条主线初始全部折叠（CSS `.gantt-children{display:none}`，仅 `.open` 时显示），首屏只呈现 6 行主线，需要细节再逐条展开——30 秒扫读场景优先。
- **统一行高 / 间距**：主线行高 40px（`gantt-track`）、子任务行高 28px（`gantt-track-child`），任务条分别为 22px / 12px 高的胶囊条，行距由网格统一控制，避免 v0.4 扁平任务条的参差。
- **紧凑化**：时间轴固定 2026-07-21 ～ 2026-08-31 共 42 天 / W1–W6 六周列，左列 250px（移动端 190px），整图 `min-width: 920px` 容器内横向滚动，不撑破页面。

## 4. Pipeline 折叠交互说明

- **分组规则**：5 个固定分组——本周重点 · P0（跨状态聚合所有 P0 项）+ 进行中 Doing / 阻塞 Blocked / 待开始 Next / 已完成 Done（按 status 分组）。P0 组与状态组有交集是刻意的：P0 组回答「这周必须盯什么」，状态组回答「每件事走到哪了」。
- **默认展开 / 折叠**：本周重点、进行中、阻塞默认展开（需要立即注意的三类）；待开始、已完成默认折叠（暂不需注意力，但点击组标题可随时展开，`aria-expanded` 与数量徽标同步）。
- **卡片字段收敛**：每张卡只保留 5 个字段——任务名（module）/ 所属主线（workstream）/ 优先级（P0/P1 badge）/ 截止（due，过期自动标红「已过期」）/ 下一步（next）。另保留一枚状态 badge 作为分组内的状态提示（P0 组跨状态必需）。pipGoal / progress / output 等描述性字段从卡片移除，数据仍保留在 `pipeline.json` 供阻塞区与导出使用。
- **搜索时强制展开**：顶栏搜索输入时全部分组强制展开并实时过滤卡片与本周待办行（数量徽标同步变为「命中数」）；清空搜索后自动恢复用户手动设定的折叠状态（`data-open-state` 记忆），不打断浏览动线。

## 5. 依赖关系图结构

图为 1 条主链路 + 3 条支线（SVG 贝塞尔曲线 + 玻璃节点，viewBox 1000×700）：

- **主链路 · 绩效交付（黄色渐变）**：绩效评估表目标 → BD计划文档 → 客户手册/FAQ → 设计交付包 → 客户 Pipeline → 注册/KYC/首单 → 周报/CRIB复盘。
- **支线 A · 获客线索汇入（浅色虚线）**：TG客户资料 / 销售转介 / 存量客户池 → 客户筛选 → 汇入「客户 Pipeline」。
- **支线 B · 看板交付与访问（淡蓝）**：看板结构整理 → 页面内容收敛 → **访问权限控制** → Simon访问 → 每周更新。
- **支线 C · 设计物料制作（浅灰）**：文字稿整理 → 提交设计团队 → UI出图 → 反馈修改 → 汇入「设计交付包」。

**「访问权限控制」节点的位置与含义**：它位于支线 B 中段，是「页面内容收敛（本轮结构收敛，Done）」与「Simon访问（Next）」之间的当前瓶颈节点（Doing · 确认方式）。含义：看板本体已就绪，只有在访问方式（Cloudflare Access / 授权邮箱）确认并配置完成后，Simon 才能真正访问，随后看板才进入「每周更新」的可持续节奏。它把原本散落在私密部署文档里的关键依赖压缩为图上一个状态化节点。

## 6. 主线任务进度（5 条线与字段口径）

05 模块渲染 `roadmap.json` 的 5 条 Workstream 主线卡（WS01–WS05）：

| id | 主线 | 字段口径 |
| --- | --- | --- |
| WS01 | 业务资料建设 | 业务手册/FAQ、BD 计划文档、设计交付包 |
| WS02 | 客户 Pipeline 建档 | TG 存量客户汇总分级与五星客户筛选 |
| WS03 | 客户转化推进 | 注册/KYC/首单（个人≥19、机构≥6、收入≥26,000 USDT） |
| WS04 | 渠道拓展 | 大数据名单、销售转介、Partner/KOL |
| WS05 | 周报与复盘 | 每周更新看板，月底 CRIB 复盘 |

- `subDone / subTotal`：关键子任务完成数 / 总数（手工维护的子任务计数，如 WS01 9/12），卡片展示「关键子任务：x/y」，是 progress 百分比之外的结构性佐证。
- `risk`：风险描述；`"无"` 时弱化显示（risk-none），有内容且主线为 Blocked 时红色强调（risk-hot + 卡片红边脉冲）。
- 进度条直接落终值、百分比 count-up；执行摘要「总体进度」= 5 条线 `progress` 的平均值。

## 8. 周更机制

- **节奏**：每周五更新一次（页面 09 模块徽标、侧栏与页脚均标注）。更新动作：
  1. 改 `data/weekly-log.json`——`updatedAt` 改为当周日期、`focus[]` 换为下周重点、`logs[]` 头部追加一条本周记录（`week` 用 ISO 周编号如 `2026-W30`）；
  2. 同步改 `todo.json` / `pipeline.json` / `gantt.json` / `roadmap.json` / `kpi.json`（按实际进展），里程碑有变化时改 `milestones.json`；
  3. 运行 `python3 .dev-scripts/sync_fallback.py` 同步 FALLBACK（看到 `VERIFY: PASS` 才算完成）；
  4. `python3 -m http.server 8080` 本地刷新预览自查（8080 被占就换端口）；
  5. `git commit && git push`。
- **FALLBACK 同步硬规则**：`app.js` 内置 FALLBACK（`__FALLBACK_SYNC_BEGIN/END__` 标记之间）必须与 7 个 JSON 逐字一致，由脚本维护、禁止手改。提交前可用 `python3 .dev-scripts/sync_fallback.py --check` 只校验不写文件，输出 `VERIFY: PASS` 才允许提交。弃用的 4 个 JSON 不参与同步。

## 9. 后续可优化项

1. **Cloudflare 私密部署落地**：依赖图「访问权限控制」节点转 Done 的前提——按 `docs/04_private_deployment.md` 完成 Pages + Access 配置，实测未登录拦截与授权邮箱放行后，把私密链接单独发给 Simon。
2. **周更记录多年化**：`logs[]` 增长到 ~12 条后可加「按月折叠」或只保留近 8 周 + 归档文档，避免时间线无限拉长。
3. **里程碑临期提示**：执行摘要「下一里程碑」距节点 ≤3 天时自动加黄色提醒样式（当前仅有状态 badge）。
4. **甘特今日线**：在 42 天时间轴上叠加 TODAY 竖线（当前 `TODAY` 常量已存在，仅用于过期判断），增强「现在走到哪」的直觉。
5. **弃用文件清理决策**：4 个弃用 JSON 若连续两轮周更无人引用，建议移入 `data/_archive/` 或删除（保留 git 历史即可），减少新维护者的认知噪音。
6. **导出增强**：当前导出仅 pipeline JSON；可考虑每周五一键导出「周报 markdown」（汇总 weeklyLog + 阻塞 + 下一里程碑），直接粘贴进周报消息。
