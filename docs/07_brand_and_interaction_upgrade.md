# 07 — 品牌与交互升级说明

> 文档版本：v1.0 ｜ 更新日期：2026-07-21 ｜ 维护人：Sera
> 适用范围：v2「品牌与交互升级」——动态图形 KPI、甘特图悬浮详情、逻辑链路图、品牌资产四件套、页面收敛中文化
> 配套文档：`docs/06_gantt_and_pipeline_structure.md`（数据结构与更新口径）、`README.md`（品牌资产替换与 KPI 维护）

本文档说明本轮（v2）品牌与交互升级的设计动机与维护方式：为什么把 KPI 从静态卡片升级为动态图形、为什么引入甘特图与逻辑链路图、品牌 Logo 的重构思路、页面中汇报对象的抽象化命名规范、六个 KPI 动态组件的设计依据，以及后续进度数据的维护流程。

---

## 1. 为什么从静态卡片升级为动态图形 KPI

v1 的 KPI 区是 8 张统一样式的静态卡片（小标题 + 大数字 + 趋势行 + done/total 细进度条）。它解决了「有没有数据」，但没有解决「数据是什么性质」：

1. **不同指标的进度性质不同，统一卡片抹平了差异**。注册人数是「累积型」指标（一个一个往上加），机构注册是「枚举型」指标（6 家逐个落定），交易收入是「金额型」指标（USDT 数字增长），设计交付包是「完备型」指标（8 项资料齐不齐），客户 Pipeline 是「转化型」指标（逐级流转），渠道拓展是「多线并计型」指标（Partner / KOL / 销售三条线合计）。用同一种卡片表达六种性质，上级只能靠读文字脑补进度形态。
2. **图形比数字先传达紧迫感**。水位只到 1/5 的水箱、6 格只亮了 0 格的分段格、几乎空着的金额条——不需要读任何文字，0.5 秒内就能感知「还差得远」。静态卡片的大数字（如 0/19）需要一次认知转换才能产生同样的判断。
3. **汇报场景是 30 秒扫读**。本看板的核心场景是 Simon 快速查看「进展如何 / 哪里卡住 / 需要什么支持」。动态图形把「进度性质 + 完成度 + 缺口」压缩进一个视觉符号，扫读效率显著高于 8 张同构卡片。
4. **收敛信息密度**。8 张卡片中「Partner / KOL / 销售对接」三张本质是同一条渠道拓展线的子项，拆成三卡是平铺、合成多轨条才是结构。收敛为 6 张动态图形卡后，一屏即可看全 PIP 核心目标。

因此 v2 的 KPI 区按指标性质分别选型：**水位（water）/ 分段格（segments）/ 金额条（money）/ 进度环（ring）/ 迷你漏斗（funnel）/ 多轨条（multi）**，每种组件只服务它最擅长表达的那类进度（各组件 rationale 见第 6 节）。

## 2. 为什么新增甘特图（Trial Timeline Gantt Chart）

（沿用 `docs/06` 第 1 节结论，此处从「上级视角」补充。）

卡片表达「状态」，不表达**时间顺序**与**并行关系**：

- **时间顺序**：PIP 试用期只有 6 周（W1 07/21 – W6 08/31），「这件事排在什么时候做、横跨哪几周、是否按计划推进」是上级判断进度健康度的第一问题。甘特图把任务条放进 W1–W6 时间轴，6 条业务主线各占一条泳道，任务条按状态着色（Done=绿 / Doing=蓝 / Next=黄 / Blocked=红），时间与状态同屏直读。
- **并行关系**：渠道拓展三条获客线、私密看板线与业务主线是并行推进的——在甘特图的泳道并置结构里，「哪些线在同时跑、哪条线空窗」一目了然；v2 任务条新增 `owner` / `next` 悬浮详情字段后，鼠标悬停即可看到「谁在做、下一步是什么」，无需跳转到其他区块。

## 3. 为什么用逻辑链路图区分串行 / 并行

甘特图擅长表达「时间」，但不擅长表达「因果」。本看板用**逻辑链路图（Workstream Dependency Map）**把任务关系拆成两类显式表达：

- **串行主链路**：设计交付包整理 → 提交设计团队 → 客户资料字段确认 → 客户唤醒 → 注册 / KYC → 首单 → CRIB 复盘。链上前置不完成，后续无法有效启动（如未完成 KYC 无法发起 OTC 首单）。串行链以单线节点图呈现，暴露**关键路径**——任何一环 Blocked，下游全部顺延。
- **并行获客线**：大数据名单、集团销售转介、Partner / KOL 拓展三条路径彼此独立、互不传染，共同汇入客户池后再进入主链路的「客户唤醒」环节。并行线以汇聚结构呈现，说明「一条线卡住不等于获客停摆」。

串行看「谁卡住谁」，并行看「互不阻塞、殊途同归」——两种结构在一张图上分区呈现，上级能同时判断「关键路径风险」与「获客纵深」。链路构成明细见 `docs/06` 第 2、3 节。

## 4. 品牌 Logo 重构思路（双焰 / 双流体基因）

v2 为看板建立自有品牌识别，替代纯文字标题。设计基因为**「双焰 / 双流体」**：

- **左焰深蓝 `#232B67`**：稳重、机构感，对应 OTC USD 大宗业务的合规与信任底色；
- **右焰亮蓝 `#18A7E3`**：流动、增长，对应 BD Pipeline 的推进动能；
- **黄色 `#FFE000` 火花点缀**：承接看板既有的黄色主视觉（主行动色 / 关键图表），落在双焰交汇处，象征「点燃转化」的动能瞬间；
- **双焰 / 双流体造型**：两簇火焰（亦可读作两股流体）相互缠绕上升，寓意「设计交付 × 客户转化」双线并进、「Sera 执行 × Simon 支持」双向协同。

**Wordmark**："HTX" 用粗字重（bold），"OTC BD" 用细字重（light / regular）紧随其后——主品牌锚定在 HTX，业务线作为次级信息，层级分明。

**产出四件套（纯 SVG 原创，无位图、无外部素材、无字体依赖，全部转曲/路径化）**：

| 文件 | 用途 |
| --- | --- |
| `assets/brand/icon.svg` | 方形图标（双焰符号，无文字），用于导航栏品牌位、社交预览 |
| `assets/brand/logo-horizontal-light.svg` | 横版 Logo（图标 + wordmark），用于日间（浅色背景）页面 |
| `assets/brand/logo-horizontal-dark.svg` | 横版 Logo（图标 + wordmark 反白/提亮版），用于夜间（深色背景）页面 |
| `assets/brand/favicon.svg` | 浏览器标签页图标（`index.html` `<head>` 中 `<link rel="icon">` 引用） |

替换与维护方式见 `README.md`「品牌资产替换」一节。

## 5. 页面中 Simon 的抽象化呈现（命名规范）

看板是向上级汇报的正式工作产物，称谓需要职业、稳定、可复制到任何场合：

- **统一使用 `Simon`**：页面、文档、数据、变量、文案中，汇报对象一律记为 `Simon`；禁止使用任何亲昵称呼或拼音变体。
- **需要抽象代称时**：可使用 **Primary Reviewer**（首要审阅人）、**Supervisor**（上级）、**Stakeholder**（相关方）等中性角色词，使语句自然（如「面向上级 Simon（Primary Reviewer）的内部业务进度 Dashboard」）。
- **可出现的同事名仅限**：Sera / Simon / 静格 / Oscar / Kimi。其他任何真实姓名不进页面、不进数据、不进文档。
- **历史记录同样适用**：CHANGELOG、TASK_STATUS 等历史条目中保留的旧称谓一并替换，保证全仓库任一时刻的任一文档都不出现旧称呼。
- **连带替换**：旧称呼的拼音变体（如环境变量占位邮箱、Basic Auth 示例用户名）同步改为 `simon@example.com` / `simon`，避免变相暴露。

## 6. 各 KPI 动态组件设计 rationale

六个 KPI 卡片按「指标性质 → 组件选型」一一对应，组件类型由 `data/kpi.json` 的 `component` 字段驱动：

| KPI | 组件（`component`） | 设计 rationale |
| --- | --- | --- |
| 个人注册 | **水箱水位**（`water`） | 人数是「累积型」指标：每多 1 人水位涨一格，19 人灌满。水箱造型直观传达「池子在蓄人」，半满状态天然带紧迫感；水位高度 = `done / total`，配合波面动效强化「在涨」的感觉 |
| 机构注册 | **6 分段格**（`segments`） | 机构是「枚举型」指标：6 家是 6 个离散目标，谈下一家就点亮一格。分段格把「N/6」从分数变成「落子」过程，空格比百分比更能催行动 |
| 交易收入 | **发光金额条**（`money`） | 收入是「金额型」指标：核心是 USDT 数值本身。金额条以发光填充表达进度（0 / 26,000），数值滚动计数，突出「钱在累积」；发光处理与品牌黄色火花呼应 |
| 设计交付包 | **进度环**（`ring`） | 交付包是「完备型」指标：8 项资料齐不齐是一个闭环问题。进度环 8/8 闭合即完成，环形自带的「圆满」语义匹配「交付包已齐」的状态（当前 `status: done`） |
| 客户 Pipeline | **迷你漏斗**（`funnel`） | Pipeline 是「转化型」指标：价值不在单一数字而在阶段流转。迷你漏斗用 3–4 级缩略漏斗呼应 07 区块的完整 CRM 漏斗（咨询 → 信息收集 → 注册 → KYC/KYB → 报价 → 首单 → 长期维护），点击/查看明细下沉到 07 区块 |
| 渠道拓展 | **多轨条**（`multi`） | 渠道是「多线并计型」指标：Partner / KOL / 销售对接三条线各有目标（≥7 / ≥10 / ≥6），合计 23。多轨条三条并行细轨各自渲染 `done / total`，顶部给合计进度，既看单线短板也看总盘 |

**交互一致性约定**：六张卡共享同一卡框（圆角、Soft Glass 毛玻璃质感、轻阴影、状态色描边），组件在卡内区域渲染；`status` 语义不变（Done=绿 / Doing=蓝 / Next=黄 / Blocked=红 / Pending=灰），组件动效只做「数据表达」，不改变状态语义。

## 7. 后续如何维护进度数据

### 7.1 更新流程（硬规则）

1. **只改 `data/*.json`**：所有进度数据与代码分离，改数据不改代码，刷新页面即生效。
2. **同步 `app.js` FALLBACK**：`app.js` 内置同名 FALLBACK 离线兜底数据，必须与 `data/*.json` **逐字一致**——改完 JSON 必须同步更新 FALLBACK 对应部分，并用比对脚本验证两者逐字相同，**不一致不允许提交**（这是 `file://` 双击打开仍能完整渲染的唯一保障，详见 `docs/06` 第 7.4 节）。
3. **本地双主题自查**：`python3 -m http.server 8080` 预览，日间 / 夜间两种主题各过一遍，确认动态组件渲染正常。
4. **脱敏自查**：无真实客户姓名、HTX UID、TG 用户名/路径、银行账户、邮箱、电话（同事名 Sera / Simon / 静格 / Oscar / Kimi 除外）。
5. `git commit && git push`。

### 7.2 `kpi.json` 字段与 `component` 取值

每个 KPI 元素字段：`label`（指标名）、`component`（组件类型）、`done`（已完成数值，组件分子）、`total`（目标数值，组件分母）、`unit`（单位）、`target`（目标文案）、`current`（当前值文案）、`trend`（趋势说明）、`next`（下一步动作）、`status`（done/doing/next/blocked）；`multi` 组件额外含 `sub[]` 子轨数组（元素为 `{ "label", "done", "total" }`）。

| `component` 值 | 渲染组件 | 适用 KPI |
| --- | --- | --- |
| `water` | 水箱水位 | 个人注册 |
| `segments` | 分段格（逐格点亮） | 机构注册 |
| `money` | 发光金额条（数值滚动） | 交易收入 |
| `ring` | 进度环 | 设计交付包 |
| `funnel` | 迷你漏斗 | 客户 Pipeline |
| `multi` | 多轨条（`sub[]` 三线） | 渠道拓展 |

### 7.3 KPI `done` / `total` 口径表（禁止凭感觉填数）

| KPI | `done` 口径 | `total` 口径 |
| --- | --- | --- |
| 个人注册 | 已完成注册人数 | 19 |
| 机构注册 | 已完成注册家数 | 6 |
| 交易收入 | 累计交易收入（USDT） | 26000 |
| 设计交付包 | 已完成交付文件数 | 8（应交付文件总数） |
| 客户 Pipeline | CRM 已建表套数 | 1 |
| 渠道拓展（合计） | `sub[]` 三线 `done` 之和 | 23（= 7 + 10 + 6） |
| ├ Partner | 已确认合作家数 | 7 |
| ├ KOL | 已确认合作位数 | 10 |
| └ 销售对接 | 已完成对接次数 | 6 |

- 渠道拓展卡的顶层 `done` / `total` 必须等于 `sub[]` 三线之和，改子线后记得同步顶层合计。
- 其余指标（Workstream 总进度、甘特任务条进度、漏斗转化率）口径见 `docs/06` 第 7.3 节。
