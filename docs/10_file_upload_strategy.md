## 1. 背景与结论

- 看板第 03 区块「资料访问中心」当前为**阶段 1 形态**：`data/resources.json` 维护资料索引（标题、说明、脱敏等级、文件路径），文件实体放在 `assets/files/`（仅脱敏文件），页面逐条渲染「打开」按钮，点击即在新标签页打开静态文件。
- 阶段 1 已满足「Simon 能查阅脱敏交付物」的需求；「在线上传新文件」属于**阶段 2**，需评估后再启动。
- **核心结论：本项目是纯静态站（无框架、无后端、无 CDN 构建），静态 HTML 无法独立完成安全上传与持久化。** 阶段 2 若启动，必须引入服务端组件（Serverless 函数 / 对象存储 / 后端服务），不存在「纯前端改一改就能上传」的路径。

## 2. 为什么纯静态站做不了安全上传

1. **没有写入端**：浏览器端 JS 只能「读」服务器上的静态文件，无法把文件「写回」 Pages / CDN。上传动作必须有一个接收文件的 HTTP 端点（服务端），静态托管不提供。
2. **没有持久化**：`localStorage` / IndexedDB 只存在访客本人浏览器里，换设备即丢失，且 Simon 上传的文件 Sera 看不到——不构成「资料中心」语义。
3. **没有可信鉴权**：任何写在前端的密钥、签名逻辑都会随源码原样下发，可被查看与伪造（与「前端 JS 假登录」同源问题，见 `docs/05_access_control.md` 第 3 节）。上传端点的鉴权必须在服务端完成。
4. **安全红线约束**：本项目数据分级要求「真实客户信息永不入库」（`SECURITY.md`）。上传功能一旦存在，就必须配套上传前脱敏校验与服务端防护，复杂度远超静态站自身能力。

## 3. 可选方案对比

| 维度 | 方案①：Cloudflare R2 + Worker ★推荐 | 方案②：Supabase Storage | 方案③：S3 兼容存储（AWS S3 / OSS / COS） | 方案④：内部文档系统链接 |
| --- | --- | --- | --- | --- |
| 形态 | R2 对象存储存文件 + Worker 提供受 Access 保护的上传/下载端点 | Supabase 托管存储 + 其 Auth/RLS 做权限 | 云厂商对象存储 + 预签名 URL 直传 | 看板只放链接，文件留公司私有云盘 / 内部文档系统 |
| 与现有架构协同 | **同生态**：站点已在 Cloudflare Pages + Access，Worker 路由可直接挂在同一 Access 应用下，访客无感 | 引入第二个平台与第二套账号/权限体系 | 引入第二家云厂商，权限体系独立 | **零开发**，沿用阶段 1 索引结构，把「文件路径」换成外链 |
| 鉴权方式 | Cloudflare Access 在边缘拦截，Policy 复用「仅 Simon 邮箱」 | Supabase Auth + Row Level Security，需自行配置且密钥管理复杂 | 需自建签发服务（又落到 Worker/后端） | 依赖内部系统自身权限（需确认 Simon 有权限） |
| 成本 | 免费额度（R2 10GB/月、Worker 10 万次请求/日）对本场景足够 | 免费档可用，超额按量 | 按量计费，另有自建端点的运维成本 | 零成本 |
| 开发量 | 中：一个 Worker（预签名 URL 签发 + 类型/大小校验） | 中：接入 SDK + RLS 策略 | 中高：签发服务 + 跨云配置 | 极低：只改 `data/resources.json` |
| 风险点 | Worker 代码需安全评审（防绕过） | 前端暴露 anon key 的设计需吃透，误配即泄露 | 预签名 URL 泄漏窗口、跨域配置复杂 | 链接外泄风险转移到内部系统；外链不可用即死链 |
| 结论 | **阶段 2 首选** | 备选（团队已用 Supabase 时） | 备选（公司已有对象存储规范时） | **阶段 1.5 兜底**：若阶段 2 迟迟不启动，先用它承载大文件 |

## 4. 推荐方案①要点（Cloudflare R2 + Worker）

1. **存储**：R2 bucket 私有，不开放任何公开读；所有读写必须经过 Worker。
2. **端点**：Worker 暴露两条路由——`POST /upload`（校验通过后签发 R2 预签名上传 URL 或直收小文件）、`GET /files/<key>`（校验通过后回源取文件）。
3. **鉴权**：Worker 路由绑定到现有 Cloudflare Access 应用（与看板同一域名/子路径），**未通过 Access 验证的请求在边缘即被拒**，Worker 内再校验 Access 签发的 JWT 做双保险。
4. **索引联动**：上传成功后由 Worker 回写或人工更新 `data/resources.json`，阶段 1 的渲染逻辑无需改动——资料中心对 Simon 的呈现方式保持一致。
5. **不可逆点**：R2 与 Pages 同属一个 Cloudflare 账号，撤销访问 = 改 Access Policy 一处即可全端失效，符合 `docs/05_access_control.md` 第 4 节的撤销策略。

## 5. 权限与安全设计（无论最终选哪个方案都必须满足）

- **上传入口同样走 Access 鉴权**：上传端点与看板同属一个 Access 应用，Policy 仍只放行 Simon 邮箱；**不允许**出现「上传页单独一套弱密码」之类的第二入口。
- **脱敏前置**：文件进入上传流程前必须人工完成脱敏（无真实客户姓名、HTX UID、TG 用户名/路径、银行账户、邮箱、电话、客户原话；占位编号 Client-XXX；同事名 Sera / Simon / 静格 / Oscar 除外）。上传界面固定展示该提示，脱敏责任不因有了上传功能而转移给工具。
- **服务端校验**：文件类型白名单（pdf / png / jpg / xlsx / csv / md）、单文件大小上限（建议 ≤ 20 MB）、文件名规范化（禁路径穿越字符）；恶意文件不落地。
- **下载最小化**：下载走短时效预签名 URL 或 Worker 鉴权回源，bucket 永不公开。
- **审计**：保留上传/下载日志（谁、何时、哪个文件），Cloudflare 侧可直接用 Access 日志 + Worker 日志。
- **撤销**：一处 Policy 变更全端生效；密钥只在 Cloudflare 环境变量，严禁进前端代码与 git（红线同 `SECURITY.md`）。
